# Governance Tiers

The framework adapts its governance level to the maturity of the organization.
Set `governance.tier` in `project-config.yaml` to control which checks, artifacts,
and validations the agents apply.

## Core Principle

**dbt is always the engine.** Tests, contracts, and enforcement run in dbt.
External standards (ODCS) are **interoperability layers** — they translate governance
artifacts into/from dbt's native format so other teams and tools can consume them.
They never replace dbt's own mechanisms.

```
                    ┌─────────────────────────────┐
                    │   External standards (ODCS)  │
                    │   = interoperability layer   │
                    │   (translate IN and OUT)      │
                    └──────────┬──────────────────┘
                               │
                    datacontract-cli
                    (import / export)
                               │
                    ┌──────────▼──────────────────┐
                    │        dbt (engine)          │
                    │                              │
                    │  contracts → schema enforce  │
                    │  tests     → quality enforce │
                    │  freshness → SLA enforce     │
                    │  meta      → classification  │
                    │                              │
                    │  SINGLE SOURCE OF TRUTH      │
                    └─────────────────────────────┘
```

## Tiers

### basic

**For:** Startups, small teams, demos, learning dbt.

| Feature | Enabled |
|---------|---------|
| Naming conventions (stg_, int_, fct_, dim_) | Yes |
| PK tests (not_null + unique) | Yes |
| FK tests (relationships) | Yes |
| Unit tests on calculated fields | Yes |
| dbt contracts on marts | No |
| meta.classification (PII) | No |
| Source freshness checks | No |
| Source schema governance (vars) | No |
| Model versioning | No |
| ODCS contracts | No |
| Attestations | No |

**Agents skip:** contract enforcement, PII checks, freshness requirements, schema
governance dimension in reviewer, versioning checks.

### standard (default)

**For:** Mid-size teams, production projects, projects that care about quality.

Everything in `basic`, plus:

| Feature | Enabled |
|---------|---------|
| dbt contracts on marts (`contract: enforced: true`) | Yes |
| meta.classification (PII/confidential/internal/public) | Yes |
| Source freshness checks | Yes |
| Source schema governance (vars, not hardcoded) | Yes |
| Model versioning (for public marts with breaking changes) | Yes |
| ODCS contracts | No |
| Attestations | No |

This is what the framework does today. No extra tools needed.

### governed

**For:** Enterprise, regulated industries (banking, insurance, healthcare),
multi-team organizations that need to share contracts with non-dbt consumers.

Everything in `standard`, plus:

| Feature | Enabled |
|---------|---------|
| ODCS contracts as interoperability layer | Yes |
| `datacontract-cli` for import/export | Yes |
| Attestation records per phase | Yes |
| SLA monitoring from ODCS servicelevels | Yes |

**How ODCS works at this tier:**

ODCS is an **input/output format**, not a replacement for dbt:

```
INPUT:  Team provides ODCS contract
          ↓
        datacontract export --format dbt → dbt YAML (sources, schemas)
          ↓
        dbt build + dbt test (dbt is the engine, always)
          ↓
OUTPUT: datacontract import --format dbt → updated ODCS contract
          ↓
        Shared with data catalog, other teams, compliance
```

If no ODCS contract is provided, the agents generate one FROM the dbt project:
```
dbt manifest.json → datacontract import → ODCS contract (auto-generated)
```

**Attestations** are Markdown records generated by the reviewer at the end of Phase 5:
```markdown
## Attestation — {feature_name}

| Check | Result | Agent | Timestamp |
|-------|--------|-------|-----------|
| Schema matches contract | PASS | dbt-reviewer | 2026-04-20T10:00:00Z |
| PII classification complete | PASS | dbt-reviewer | 2026-04-20T10:00:00Z |
| Test coverage ≥ 80% | PASS | dbt-reviewer | 2026-04-20T10:00:00Z |
| Source freshness configured | PASS | dbt-reviewer | 2026-04-20T10:00:00Z |
| Breaking changes versioned | N/A | dbt-reviewer | 2026-04-20T10:00:00Z |
```

### enterprise

**For:** Data mesh, multi-project, organizations with centralized metrics governance.

Everything in `governed`, plus:

| Feature | Enabled |
|---------|---------|
| Centralized metrics library | Yes |
| Cross-project contract validation | Yes |
| Mandatory model versioning on all public models | Yes |

**Metrics library:** A shared repository of canonical metric definitions that projects
reference. The `dbt-semantic` agent validates that metric implementations match the
library definitions. Prevents metric drift (5 teams calculating NPL 5 different ways).

**Cross-project validation:** The `dbt-reviewer` checks that cross-project `ref()`s
target versioned, contracted models. The `dbt-ops` agent monitors cross-project
health via MCP (requires `all_projects: true` on the service token).

> **Note:** Enterprise tier requires dbt Mesh architecture and multi-project setup.
> The framework's Mesh support is planned but not yet fully implemented.

## How Agents Use Tiers

Each agent reads `governance.tier` from `project-config.yaml` at the start:

```
tier = read project-config.yaml → governance.tier (default: "standard")
```

**Behavior per tier:**

| Tier check | Agent behavior |
|-----------|---------------|
| `tier >= standard` | Enforce contracts, PII classification, freshness, schema governance |
| `tier >= governed` | Check for ODCS contract, use datacontract-cli, generate attestations |
| `tier >= enterprise` | Validate against metrics library, cross-project checks |

Agents at lower tiers **skip** higher-tier checks silently — no warnings, no noise.
The user chose the tier; the framework respects it.

## Tool Dependencies per Tier

| Tier | Required tools | Optional tools |
|------|---------------|----------------|
| basic | dbt | — |
| standard | dbt | — |
| governed | dbt, datacontract-cli (`pip install datacontract-cli`) | — |
| enterprise | dbt, datacontract-cli | metrics library repo |
